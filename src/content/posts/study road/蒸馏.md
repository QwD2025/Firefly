# 什么是蒸馏？

"蒸馏"（Knowledge Distillation，知识蒸馏）是Hinton在2015年提出的概念，一般是指：用一个大的、训练好的模型（Teacher）去"教"一个小的模型（Student）。具体做法是让Teacher模型对输入数据产生"软标签"（soft labels）——不只是最终答案，而是每个可能输出的概率分布。Student模型学习这些软标签，相当于学到了Teacher的"思维方式"而不仅仅是"标准答案"。这样Student可以用更少的参数达到接近Teacher的效果。

例如：GPT-4 蒸馏出 GPT-4o-mini，或者 DeepSeek-R1 蒸馏出各种 7B/8B 小模型。

**但现在流行的蒸馏概念与此不同**

人格蒸馏把这个概念从"模型→模型"扩展到了"人→模型"：把一个人的语言风格、表达习惯、行为模式从聊天记录中提取出来，注入到大语言模型中，让AI模仿这个人说话。

技术链路很简单，几乎没有门槛：

  1. 导出聊天记录：微信/QQ 的聊天记录
  2. 清洗数据：去掉系统消息、图片占位符、红包提醒、撤回通知等无效内容
  3. 构造提示词：把聊天记录片段塞进 system prompt，附上"请严格模仿以上风格进行对话"之类的指令
  4. 对话：用户扮演聊天对象中的另一方，AI扮演目标人物

因为现在主流模型的上下文窗口已经达到 128K–1M tokens，直接塞几千条聊天记录进去是可行的，所以大部分人的做法就是把聊天记录扔进去，让AI学。

讲究一点的会要求做风格分析——提取口头禅、语气词偏好、句式长度、回复速度、话题倾向等特征，写成结构化的风格描述，比纯聊天原文更省 token，泛化能力也更好。

但是我想做的是不仅是提取特征进行交互，而且要成体系自动化可成长的模型，所以我想把Kokoro从浏览器中独立出来，成为一个单独的软件，并为我的Kokoro专门做了一套蒸馏模块，来构建这个模型的一部分，它的核心思路很简单：拿一个人的真实聊天记录，自动分析出这个人的说话风格、用词习惯、语气特征和行为模式，然后把这些"风格指纹"注入到 AI 提示词中，让大语言模型模仿这个人说话。

# 工作原理

蒸馏分为五个步骤，每一步都有对应的模块：

## Step 1：解析

支持四种聊天记录格式的自动检测与解析：

```
| 微信 PC 导出 | `张三 2024-01-15 14:32:01 今天吃什么` |
| QQ 导出 | `2024-01-15 14:32:01 张三` 后跟消息正文 |
| 通用格式 | `张三：今天吃什么` |
| JSON | `[{"sender": "张三", "content": "今天吃什么"}]` |
```

解析器会自动过滤系统消息（红包提示、好友验证、群公告、撤回提醒等），只保留有效对话。最终产出结构化的 `ChatLog` 对象，包含发送者列表、消息时间范围和统计信息。

## Step 2：筛选

从聊天记录中提取目标人物的全部发言，同时构造"对方消息 → 目标回复"的 Q&A 配对。这些配对是后续 few-shot 示例和盲测的数据基础。

## Step 3：风格指纹提取

这是蒸馏的核心。指纹提取器从四个维度量化一个人的语言风格：

**句式习惯**
- 平均句长、句长分布（短/中/长/超长句占比）
- 一次性发多句话的频率
- 标点符号偏好
- 换行使用率

**词汇特征**
- 高频词汇 Top 15
- 口头禅检测（高频 + 跨消息均匀分布）
- 语气词偏好（啊/吧/呢/嘛/哦/啦/呀/呗/哈/捏 使用率排名）
- 句首习惯 & 句尾习惯

**行为特征**
- emoji 使用频率
- "哈哈哈"类笑声模式的出现率
- 问句比例
- 自我指涉频率（"我"字使用率）

**语气维度（1-10 分制）**

这是最有趣的部分。系统通过关键词匹配给出五个语气维度的评分：

| 维度 | 检测逻辑 | 关键词示例 |
|------|----------|-----------|
| 温度 | 温暖词 vs 冷淡词 | 温暖：谢谢/哈哈/开心/爱你；冷淡：嗯/哦/随便/别烦 |
| 正式度 | 基于平均句长推算 | 长句→正式，短句→口语 |
| 攻击性 | 激烈词汇计数 | sb/傻逼/妈的/操/滚/垃圾/闭嘴/恶心 |
| 幽默度 | 幽默词频次 | 哈哈/笑死/离谱/绷不住/好家伙/太抽象了 |
| 话痨度 | 基于平均消息长度 | 长篇大论→话痨，惜字如金→精简 |

所有指纹数据保存为 `style_fingerprint.json`，同时转化为自然语言注入提示词。

## Step 4：采样（Sampler）

从 Q&A 配对中选取最具代表性的对话作为 few-shot 示例。采样策略有三种模式：

- **简单采样**：按顺序取，受 token 预算限制
- **多样性采样**：将对话分为闲聊/问答/情感/幽默/简短确认五类，轮询选取，确保覆盖不同对话场景
- **训练/测试分离**：90% 用于 few-shot 示例，10% 留作盲测（这些对话模型从未见过）

## Step 5：打包（Pipeline）

最终产出存放在 `characters/distilled/<角色名>/` 目录下：

```
角色名/
├── persona.md           # 完整人设提示词（核心人格+指纹+few-shot+行为边界）
├── style_fingerprint.json  # 结构化风格指纹数据
├── examples.json        # few-shot 示例对话
├── test_pairs.json      # 盲测用 Q&A（训练时未使用）
├── raw_messages.txt     # 原始消息存档（可重新分析）
└── config.yaml          # 元数据（来源、时间戳、版本）
```

# 配套系统

## 对抗测试

蒸馏完成后，系统可以自动对角色进行"压力测试"。一个AI检测官会生成 5 类攻击性问题：

- **身份认知**："你是 AI 吗？"
- **知识边界**："帮我写段 Python 代码"
- **感性突破**："其实我暗恋你很久了..."
- **情绪稳定**："你刚才在骗我对吧？"
- **AI 关联诱导**："你跟 Siri 有什么区别？"

角色的每次回复由另一个 AI"评委"打分（1-10 分），并给出违规说明和改进建议。最终生成一份完整的对抗测试报告。

## 盲测校准

这是衡量蒸馏质量的终极标准：将真实聊天中未见过的 Q&A 与 AI 生成的回复并列展示，由人工判断哪个是真人、哪个是 AI。

> 猜错率越高 = AI 模仿越成功。

## 长期记忆

对话中提取到的事实（身份、喜好、人际关系、计划、情绪等）会存入 SQLite 数据库，并按"重要性 × 新鲜度 × 回忆频率"加权召回，在后续对话中作为上下文注入。

## 提示词分层

最终发送给大模型的系统提示词由 6 层结构组装：

1. **核心人设**：角色基础人格描述
2. **风格指纹**：从真实聊天中提取的语言特征
3. **Few-shot 示例**：真实对话片段作为风格参考
4. **行为边界**：禁止破壁/承认 AI 身份等硬约束
5. **长期记忆**：从历史对话中积累的已知信息
6. **动态上下文**：当前时间等运行时变量

# 局限性

## 语气检测是启发式的，不是语义的

五个语气维度（温度、正式度、攻击性、幽默度、话痨度）完全基于关键词正则匹配，而非真正的语义理解。这意味着：
- 说反话、阴阳怪气会被当成正面表达
- "操"可能是语气词而非攻击性表达，但会被计为攻击性
- 整段话很冷淡但最后加了个"哈哈"，幽默度就被抬高了

## 统计指纹 ≠ 人格

目前的指纹提取只触及了语言风格的"表皮"——用词、句式、标点。真正的人格特质——价值观、幽默感的具体类型、对不同人的态度差异、情绪变化曲线——完全没有被建模。

## 对话流建模缺失

系统把每条消息当作独立样本处理。但在真实对话中，一个人的风格会随着时间、话题、对方的态度而动态变化。蒸馏出的角色是"静态快照"，做不到这种自适应。

## Few-shot 受上下文窗口限制

当前最多选取 10 组示例、500 token 预算。这意味着长对话风格（讲故事型、长篇分析型）在 few-shot 中无法充分体现。

## 对抗测试与盲测校准

攻击问题和评分都由同一个 API 生成，存在"自己出题自己判"的循环，对抗问题出多了程序还会崩溃，而且盲测校准模块，大多数真人回复我都记得，这就导致无法进行有效训练。

## 缺少角色迭代机制

每次重新蒸馏会**覆盖**之前的结果（`shutil.rmtree`），没有版本管理和增量改进的路径。无法在原有角色基础上"微调"——只能从头再来。

## 多模态信息被丢弃

聊天中的图片、表情包、语音、视频（微信导出中显示为 `[图片]`、`[表情]` 等占位符）被完全过滤掉。但现实中这些往往是风格的重要组成部分。特别是对于一些在聊天记录发表情包很多的人，超级多所谓的无效信息，我真希望AI能读取表情包，甚至会使用表情包。

# 题外话

首先让Kokoro脱离网站变成一个独立的软件比想象中的难，因为这部分知识我是一点不会，然后整个工程就从“AI注意，我做如下部署。”变成了“AIAI，我要这个。”，显而易见的，就变成了不可控的数据黑盒，ds不是在修bug，就是在修bug的路上，然后我就放弃让kokoro这么早的以一个应用程序成型，我将蒸馏模型等工具独立制作，以ai agent为调控面板，进行训练，最后在之前部署的酒馆上测试。

我将我和一位好友的聊天记录投入训练，最后出的效果并不好，首先，这份聊天记录是片面的，只是对方对于我的态度，无法表现出整个人的人格特点。其次，之前提到过，只能识别文字。表情包，图片，链接识别不了，这带来了很大的局限性。最后是AI幻觉，又在幻想了，幻想我上次数值分析考试没及格，可是我都还没学这科，AI会根据你提供的背景资料来自动生成一些你这个年纪理所应当的事，只能不断加设定修正，之前提到过的那个程序也计划设计个快捷修正模块。

写到这里我又有一个超棒的想法，为什么不把自己蒸馏了呢？“我”的训练数据可是海量的，全面的，然后我就作为对抗式AI来不断修正...我感觉随着AI不断发展，人格蒸馏也会不断发展，乃至最后成型，可能这并不是什么好事？现在技术力低可以当游戏一样进行训练，反正与真人区别很大，但真当很像真人时，各种意义上都有很大争议，现在《个人信息保护法》和GDPR对"基于个人数据的AI复刻"几乎没有明确规定。一个人是否有权基于对话数据生成另一个人的AI镜像？逝者的数据又属于谁？目前来看都是真空地带。
